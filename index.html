🧮 Was ist ein Framework?

A framework provides a comprehensive structure for your application. It dictates the architecture and workflow; you build your application according to the framework's rules and guidelines. Frameworks often come with a predefined way to organize your code, templates, data models, and sometimes even dictate how to handle certain tasks like user input or database interactions.

Ganz klar ist die Unterscheidung zwischen Framework und Library oft nicht - oder - ist schwimmend.

🤔 Welche Probleme schuf das, die moderne Frameworks lösen wollen?

🌳 Der DOM

🐛 Probleme:

  • Unmaintainable - in einer großen CodeBase ist irgendwann so viel Logik und Code drin, dass es fast unvermeidbar ist, dass man sich Seiteneffekte einfängt
  • Unperformant - Jede Anfrage im Code muss den Baum durchlaufen. D.h. wenn man viele Skripte hat, die zum Teil auch noch den Baum verändern, treten oft wieder Bugs auf und das ständige ein- und aussetzen von Nodes ist für den Browser teuer

💡 Die Lösung - Data-Driven State-Changes

Beispiel React HTML / DOM


						<!DOCTYPE html>
						<html>
						<head>
							<meta charset="utf-8">
							<meta name="viewport" content="width=device-width">
							<title>Frontendseitiges Client-HTML</title>
						</head>
						<body>
							<div id="root"></div>
						</body>
						</html>

Man sieht hier, dass es nur noch eine Stelle gibt, an der für den Browser Nodes ausgetauscht werden: Den sogenannten React-Root.

React löst das, indem sie einen ShadowDOM erstellen, also genau das, was der Browser nativ als DOM hat, nur geklont und intern mit eigenem State befüllt. Das löst die Performance Probleme größtenteils, da der DOM nicht mehr durchgehangelt werden muss im Browser (was teuer ist) und auch das ersetzen passiert immer zur Renderzeit auf einen Schlag, und nicht mehr willkürlich.
